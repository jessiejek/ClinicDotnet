## 5. Medium Finding Results (one per item)

### Item 1: Drug Interactions (MEDIUM)
**Status:** ✅
**What was done:**
- BE: Updated XML doc comments on both `AllergyCheck` and `Check` endpoints with detailed documentation explaining:
  - This is a deliberate stub — drug interaction DB is not configured
  - The frontend runs local client-side cross-referencing via `AllergyWarningBannerComponent` and `DrugInteractionService`
  - `unavailable=true` is explicitly documented as the correct signal for prescribers to verify manually
- FE: Verified the `drug-interaction.service.ts` handles `unavailable: true` via `normalizeAllergyResponse` and `normalizeInteractionResponse` — both correctly propagate the `unavailable` flag. The `AllergyWarningBannerComponent` does local cross-referencing and the `DrugInteractionService` has local drug-class conflict detection as a safety net.

### Item 2: Doctor Photo Upload (MEDIUM)
**Status:** ✅
**What was done:**
- BE: Added `POST api/doctors/{id:guid}/photo` endpoint in `DoctorsController`:
  - Injects `IFileUploadService` for file saving
  - Validates: non-empty, max 5MB, image extensions only (.jpg/.jpeg/.png/.gif/.webp)
  - Saves via `IFileUploadService.UploadAsync(fileStream, fileName, "doctor-photos", ct)`
  - Gets full URL via `IFileUploadService.GetFileUrl(relativePath)`
  - Updates doctor's `ProfilePhotoUrl` in DB via `IClinicDoctorsService.SetDoctorPhotoUrlAsync`
  - Returns `{ "profilePhotoUrl": "https://..." }`
  - Requires `Admin` role
  - Uses `[RequestSizeLimit(5 * 1024 * 1024)]`
- BE: Added `SetDoctorPhotoUrlAsync` to `IClinicDoctorsService` interface and implemented in `DoctorsService`
- FE: Updated `doctor-form.page.ts`:
  - Removed "Photo upload is mock only for this phase." text
  - Added `(change)="onPhotoSelected($event)"` on the file input
  - Added `onPhotoSelected()` method that:
    - Validates file type (image/jpeg, image/png, image/gif, image/webp)
    - Validates file size (max 5MB)
    - Creates `FormData` with the file
    - Calls `apiService.postFormData('doctors/{id}/photo', formData)`
    - Updates the local `currentDoctor.profilePhotoUrl` on success
    - Shows toast on error

### Item 3: Patient Payment/Receipt Visibility (MEDIUM)
**Status:** ✅
**What was done:**
- BE: Updated `PaymentsController`:
  - `GET api/payments/booking/{bookingId}`: `[Authorize("Admin,Staff")]` → `[Authorize("Admin,Staff,Patient")]`
  - `GET api/payments/{id}/receipt`: `[Authorize("Admin,Staff")]` → `[Authorize("Admin,Staff,Patient")]`
  - Both endpoints now pass `User` (ClaimsPrincipal) to the service methods
- BE: Updated `IClinicPaymentsService`:
  - `GetPaymentByBookingAsync` now takes `ClaimsPrincipal principal` parameter
  - `GetReceiptAsync` now takes `ClaimsPrincipal principal` parameter
- BE: Updated `BookingsService` (implements `IClinicPaymentsService`):
  - `GetPaymentByBookingAsync`: If user is Patient role, loads the booking and verifies `booking.PatientId` matches the current patient (via `GetCurrentPatientAsync`). Admin/Staff bypass ownership check.
  - `GetReceiptAsync`: If user is Patient role, verifies `payment.Booking.PatientId` matches the current patient. Admin/Staff bypass ownership check.
  - Admin/Staff retain full access — no restrictions.

### Item 4: Booking Status Naming (MEDIUM — Audit)
**Status:** ✅ **AUDITED — No mismatches found**
**Verification:**
- BE constants (BookingsService.cs): `Pending`, `ProofSubmitted`, `Confirmed`, `CheckedIn`, `InProgress`, `OnHold`, `Cancelled`, `Completed`, `Expired`, `NoShow`, `Rescheduled`
- FE types (booking.models.ts): `'Pending' | 'ProofSubmitted' | 'Confirmed' | 'CheckedIn' | 'InProgress' | 'OnHold' | 'Cancelled' | 'Completed' | 'Expired' | 'NoShow' | 'Rescheduled'`
- DB default constraint: `DF_Bookings_Status DEFAULT (N'Pending')`
- **No mismatches found.** All three layers use identical values. No mapping needed.
- Note: The `PaymentMode` default in the DB is `'Online'` but the code defaults to `PayAtClinic` — this is by design (the DB default is a schema default, while the app logic chooses `PayAtClinic` for walk-ins). No action needed.
